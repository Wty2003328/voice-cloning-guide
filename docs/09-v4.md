# 09 — v4: LoRA Fine-Tuning + 48 kHz Output

The rest of this guide focuses on **v2** — the most-used, easiest-to-debug version. This document covers **v4**, which produces noticeably cleaner audio at 48 kHz and is the recommended path if your training data is clean.

By the end you'll have v4 SoVITS LoRA + GPT trained on the same data, generating 48 kHz output with a separate vocoder pass.

## Why v4

| Aspect | v2 | v4 |
|---|---|---|
| S2 architecture | VITS-based (text/posterior encoders + flow + HiFi-GAN decoder) | Conditional Flow Matching DiT + separate vocoder |
| Sample rate | 32 kHz | **48 kHz** |
| Vocoder | Built into SoVITS decoder | Separate `vocoder.pth` (custom-trained) |
| Fine-tune | Full network update | **LoRA only** — adapter on CFM attention |
| Training loss | mel + KL + adversarial + feature matching | Single CFM loss |
| Discriminator? | Yes (MultiPeriodDiscriminator) | No |
| Wiki SIM benchmark | 0.549 | **0.735** |
| Behavior | Averages across training set — robust to dirty data | Stays close to reference audio — **needs clean data** |
| Adapter size | N/A (full ckpt ~95 MB) | LoRA ~10-50 MB |

**Pick v4 when**: your training data is clean, you want the highest quality, and you're comfortable with one extra inference stage.

**Stick with v2 when**: your data has any noise/BGM/reverb (v4 will reproduce it), you need to avoid the LoRA dependency (PEFT), or you want simpler architecture for debugging.

## v4 architecture

```
Text  →  GPT (s1, same as v2)  →  Semantic tokens  →  SoVITS DiT (s2 v4)  →  Mel (100ch)
                                                              ↓
                                                       Reference audio
                                                       (mel + features)
                                                              ↓
                                                  vocoder.pth (separate net)
                                                              ↓
                                                       48 kHz waveform
```

The DiT replaces v2's flow + HiFi-GAN with a flow-matching diffusion model. Internally, it iteratively refines a 100-channel mel spectrogram over `sample_steps` denoising steps (default 32). The output mel is then handed to the separate vocoder (a ~57 MB HiFi-GAN-style network) for final waveform synthesis.

The S1 (GPT) is architecturally identical to v2 — same Text2SemanticDecoder, same 1024-vocab semantic tokens. Only the pretrained weights differ: v4 uses `s1v3.ckpt` (trained on a larger 7,000-hour corpus vs v2's 5,000).

## Why LoRA (and what it means)

The v4 SoVITS DiT is ~700 MB — much larger than v2's ~95 MB — and full fine-tuning is impractical on consumer GPUs. Upstream solves this with LoRA: low-rank adapters on the CFM's attention layers (`to_k`, `to_q`, `to_v`, `to_out.0`).

In our setup:
- Rank 32 LoRA adapters add ~10-15 MB of trainable parameters.
- ~99% of the base model is frozen.
- Training is faster and uses much less VRAM than full fine-tune.
- Adapters can be **merged into the base** for inference (`merge_and_unload()`), so deployment doesn't need PEFT at runtime.

The downside: LoRA can only update a small subspace of the model. If your speaker is wildly different from anything in the pretrained data, LoRA may underperform a full fine-tune. For typical cases (one specific speaker, similar to general human speech), LoRA is sufficient.

## Training v4

The data pipeline is **identical** to v2 — steps 1-4 produce the same files (`2-name2text.txt`, `4-cnhubert/`, `5-wav32k/`, `6-name2semantic.tsv`) which v4 reads as-is. **Don't re-run the data prep**; just train SoVITS and GPT on existing data.

```bash
# SoVITS LoRA fine-tune (~25 min on RTX 5080)
python 05_train_sovits_v4.py --exp my_speaker --epochs 20 --batch-size 2 --lora-rank 32

# GPT fine-tune (same as v2, just point at the v3-base pretrained)
python 06_train_gpt.py --exp my_speaker --pretrained-version v4 --epochs 15
```

Output:
- SoVITS LoRA-merged checkpoints → `GPT-SoVITS/SoVITS_weights_v4/<exp>_e<N>_s<step>_l<rank>.pth`
- GPT checkpoints → `GPT-SoVITS/GPT_weights_v3/<exp>-e<N>.ckpt`

The directory naming follows upstream convention: v3 and v4 GPT share `s1v3.ckpt` as base, so the GPT output dir is named `GPT_weights_v3` regardless of which SoVITS version you pair it with.

### Hyperparameters that matter for v4

| Knob | Default | Effect |
|---|---|---|
| `--batch-size` | 2 | DiT is heavier than VITS — 2 fits comfortably on 16 GB with grad checkpointing. Drop to 1 if you OOM. |
| `--epochs` | 20 | Same as v2 default. CFM loss should drop to ~0.05-0.1 by E15+. |
| `--lora-rank` | 32 | Larger rank (64) = more capacity, more memory. Smaller (8, 16) = lighter adapters. 32 is the upstream default. |
| `--no-grad-ckpt` | off | Gradient checkpointing trades compute for memory; turn it off only if you have a lot of VRAM. |

### What healthy v4 training looks like

```
=== Epoch 1: avg_cfm_loss=0.4123 (75s) ===
=== Epoch 5: avg_cfm_loss=0.1812 (75s) ===
=== Epoch 10: avg_cfm_loss=0.0894 (75s) ===
=== Epoch 15: avg_cfm_loss=0.0628 (75s) ===
=== Epoch 20: avg_cfm_loss=0.0604 (75s) ===
```

Per-batch loss oscillates between 0.001 and 0.5 — that's expected for flow matching (different time-step samples have different difficulty). The epoch average is the signal to watch.

## Inference v4

```bash
python 07_inference_v4.py \
    --exp my_speaker \
    --text "Hello! I'm your character." \
    --lang en \
    --ref-wav ../GPT-SoVITS/logs/my_speaker/0_sliced/0003.wav \
    --ref-text "ここは私に任せて私を選んでくれる" \
    --ref-lang ja \
    --out hello_v4.wav
```

Output is **48 kHz** (vs v2's 32 kHz). File size is ~50% larger for the same duration.

### v4-specific inference flags

| Flag | Default | Effect |
|---|---|---|
| `--sample-steps` | 32 | CFM denoising steps. 32 = best quality, 16 = ~2× faster, 8 = zero-shot quality, 4 = bare minimum. Halve for faster preview, full for final renders. |
| `--lora-rank` | 32 | Must match training rank. Mismatched ranks fail to load LoRA weights. |

The other flags (`--top-k`, `--temperature`, `--ref-wav`, etc.) work identically to v2.

### Performance

Inference is slower than v2 because of the iterative CFM steps + separate vocoder pass. On RTX 5080 with `--sample-steps 32`, expect ~2-4 seconds per sentence. With `--sample-steps 8`, closer to ~0.8-1.5 seconds.

For real-time use, set `--sample-steps 16` and benchmark — the quality drop from 32 → 16 is usually not noticeable on small fine-tunes.

## When v4 underperforms v2

The wiki notes: *"v3/v4 are more faithful to the reference audio. For training sets with average audio quality, v1/v2 can deliver decent results, but v3/v4 cannot."*

Concretely:
- If your training audio has BGM bleed, v4 will reproduce it (v2's averaging-across-dataset would have washed it out).
- If your reference clip has any noise, that noise gets into every output sample.
- If your reference is very short (~3 sec) or atypical, v4 may sound less stable.

Mitigations:
- Run Demucs more aggressively — try the `htdemucs_ft` model variant for cleaner separation.
- Pick a reference clip from the *middle* of training data (not the first or last slice).
- Re-train with `--lora-rank 64` and more epochs if the speaker is hard to capture.

## Comparing v2 vs v4 on the same data

If you're unsure which to publish, train both and A/B test. Reuse the same data:

```bash
# v2 path
python 05_train_sovits.py --exp my_speaker --epochs 20
python 06_train_gpt.py --exp my_speaker --pretrained-version v2

# v4 path (same data)
python 05_train_sovits_v4.py --exp my_speaker --epochs 20 --lora-rank 32
python 06_train_gpt.py --exp my_speaker --pretrained-version v4

# Generate matching samples
python 07_inference.py    --exp my_speaker --text "..." [...] --out v2_test.wav
python 07_inference_v4.py --exp my_speaker --text "..." [...] --out v4_test.wav
```

Listen for: high-frequency clarity (v4 wins on clean data), naturalness of prosody, voice similarity to your reference, and stability across sentences. v4 typically wins on the first three; v2 sometimes wins on stability with imperfect data.
